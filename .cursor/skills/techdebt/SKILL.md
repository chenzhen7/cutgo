# /techdebt

用于每次收工前做一次低风险技术债清理，重点消灭重复代码和明显垃圾代码。

## 目标

- 优先处理“重复实现、可抽取共用逻辑、死代码、无意义分支、低价值注释、不规范的变量命名”
- 保持行为不变，不做需求扩展
- 小步提交，确保每一轮都可验证

## 执行流程

每次触发本技能时，按以下顺序执行：

1. 扫描可疑区域（重复代码 + 死代码 + 命名混乱）
2. 选择 1-3 个低风险点进行清理（不要一口气大重构）
3. 实施最小改动，优先抽公共函数/组件，删除未使用代码，更正不规范的变量命名
4. 运行必要校验（至少 lint 或类型检查；能跑测试就跑）
5. 输出“本轮还债报告”（见模板）

## 扫描策略

优先从最近改动文件开始，再扩展到同目录：

- 查重复片段：相似 if/else、复制粘贴的映射转换、重复请求/错误处理
- 查死代码：未使用的导入、函数、状态、常量、分支
- 查坏味道：超长函数、重复字符串字面量、职责混杂、命名不规范（词不达意、拼写错误、格式不统一）

推荐工具顺序：

- `rg` 精确搜关键词和重复片段
- 若范围较大，用语义搜索定位“相同意图不同实现”
- 仅读取必要文件，避免全仓大扫荡

## 改动边界（必须遵守）

- 不改变对外行为和接口契约
- 不顺手改 unrelated 文件
- 不引入新的大型抽象层
- 不修改与本次清理无关的配置
- 若发现高风险点，仅记录到“后续债务清单”，不在本轮处理

## 优先级规则

按以下优先级选择清理项：

1. 同文件内重复逻辑（收益高、风险低）
2. 同目录下重复逻辑（可提炼工具函数）
3. 明确未使用代码（可直接删除）
4. 局部变量/函数命名更正（低风险，提升可读性）
5. 复杂但高风险的历史包袱（只记录，不强拆）

## 本轮还债报告模板

每轮结束必须输出：

```markdown
## Tech Debt Sweep

- 范围：<目录/模块>
- 清理项：
  - <改动 1>
  - <改动 2>
- 风险评估：<低/中/高，及原因>
- 验证：
  - <lint/typecheck/test 命令与结果>
- 后续债务清单：
  - <本轮未处理但建议后续处理项>
```

## 默认口径

- 用户说“收工前清一下”“还技术债”“去重一下”“跑 /techdebt”时自动应用本技能
- 默认执行“低风险、小步、可验证”策略
- 如果仓库本身已存在未提交改动，只在用户当前任务相关范围内操作
