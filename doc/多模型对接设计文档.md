## 1. 背景与目标

CutGo（秒剧）是一套 AI 驱动的短剧工业流水线工具，涉及**文本生成**（剧本/大纲/角色分析）、**图像生成**（分镜画面/角色头像）、**视频生成**（动态帧/视频合成）、**语音合成**（TTS 配音）四大 AI 能力域。

直接硬编码对接每一家 SDK 会导致：

- 代码高度耦合，模型迭代成本极高
- 无法灵活切换/降级/A-B 测试
- 错误处理与重试逻辑分散，难以统一监控

**目标**：在 `lib/ai/` 下建立统一的 AI 抽象层（类型、配置读取、Provider 工厂函数），对上层业务逐步屏蔽底层模型差异。

**实现现状（与代码对齐）**：

| 能力域 | `lib/ai` 抽象层 | 业务 API 实际调用 |
|--------|-----------------|-------------------|
| **LLM** | 已实现 `getLLMProvider` / `createLLMProviderFromConfig`（Vercel AI SDK） | 多数功能仍走 **环境变量** `OPENAI_API_KEY` / `OPENAI_BASE_URL` / `OPENAI_MODEL` + 原生 `fetch('/chat/completions')` |
| **图像** | 仅 **占位** `PlaceholderImageProvider`（placehold.co） | 未接真实生图 API |
| **视频** | 仅 **占位** `PlaceholderVideoProvider` | 未接真实视频 API |
| **TTS** | 无独立 Provider 模块；设置页有选项与连通性测试 | 未在 `lib/ai` 中实现合成逻辑 |

设置页中的多模型配置（`AIModelConfig` + `Settings.active*ConfigId`）已落地，**连通性测试**走 `lib/ai`；**剧本/大纲/分析等生成链路尚未统一改用数据库中的 LLM 配置**。

## 2. 整体架构

```
业务模块（剧本/分镜/画面/TTS）
        │
        ├─ 当前主路径：部分 API Route 直接使用 env + fetch('/chat/completions')
        │
        └─ 统一层：lib/ai
                │
                ▼
         getActiveConfig(type)     ← Prisma：Settings + AIModelConfig
                │
                ▼
    getLLMProvider / getImageProvider / getVideoProvider
                │
    ┌───────────┼───────────────┐
    ▼           ▼               ▼
 OpenAILLM   GoogleLLM    PlaceholderImage / PlaceholderVideo
 Provider    Provider     （图像/视频暂无真实实现）
```

- **厂商与默认 Base URL、下拉模型列表**：集中在 `lib/ai/providers.ts` 的 `AI_PROVIDER_OPTIONS_BY_TYPE`，供设置页使用。
- **无** 单独的 `AIProviderFactory` 类；采用 **`getXxxProvider` 函数** 模式（按需即时创建）。

## 3. 即将对接的常用模型厂商一览

以下为 CutGo 计划或已预置 UI 的厂商，按能力域分类。LLM 侧与 `AI_PROVIDER_OPTIONS_BY_TYPE.llm` 中选项一致（模型 ID 以设置页为准）。

### 3.1 文本大模型（LLM）

| 厂商 | `provider` 标识 | 说明 |
|------|-------------------|------|
| **OpenAI** | `openai` | GPT 系列，OpenAI 官方或兼容网关 |
| **DeepSeek** | `deepseek` | deepseek-chat / deepseek-reasoner 等 |
| **Anthropic** | `anthropic` | 在代码中与 OpenAI 兼容通道共用同一 Provider，**需 Base URL 指向 OpenAI 兼容形态的服务**（如官方兼容代理或中转） |
| **阿里通义千问** | `qwen` | DashScope 兼容模式 Base URL |
| **字节豆包（火山方舟）** | `doubao` | 方舟 OpenAI 兼容接口 |
| **Google Gemini** | `google` | 独立实现 `GoogleLLMProvider`（`@ai-sdk/google`），非 OpenAI 兼容路径 |
| **智谱 GLM** | `zhipu` | 开放平台 OpenAI 兼容 `/v4` |
| **其他** | `custom-openai` | 自定义模型 ID + Base URL |

### 3.2 图像生成

预置：`openai`（DALL-E 3）、`comfyui`、`stability`。`lib/ai/image` 中真实 Provider **尚未实现**，当前一律返回占位图 URL。

### 3.3 视频生成

预置：`runway`、`kling`、`minimax`。`lib/ai/video` 中真实 Provider **尚未实现**。

### 3.4 语音合成（TTS）

预置：`edge-tts`、`openai`、`elevenlabs`、`minimax`。连通性测试见 `app/api/settings/ai-configs/test/route.ts`；**无** `getTTSProvider` 封装。

## 4. 标准接口定义

> 路径：`lib/ai/types.ts`

### 4.1 配置类型

```typescript
export interface AIModelConfig {
  id: string
  name: string
  type: "llm" | "image" | "video" | "tts"
  provider: string
  model: string
  apiKey: string
  baseUrl: string
  config?: Record<string, unknown>
  isDefault: boolean
}
```

### 4.2 文本大模型（LLM）

```typescript
export type LLMMessageRole = "system" | "user" | "assistant"

export interface LLMMessage {
  role: LLMMessageRole
  content: string
}

export interface LLMGenerateOptions {
  messages: LLMMessage[]
  model?: string
  maxTokens?: number
}

export interface LLMUsage {
  promptTokens?: number
  completionTokens?: number
}

export interface LLMGenerateResult {
  content: string
  usage?: LLMUsage
}

/** 当前接口为单次非流式 chat；流式输出尚未抽象到该接口 */
export interface LLMProvider {
  id: string
  chat(options: LLMGenerateOptions): Promise<LLMGenerateResult>
}
```

### 4.3 图像生成

```typescript
export interface ImageGenerateOptions {
  prompt: string
  negativePrompt?: string
  width: number
  height: number
  stylePreset?: string
  numOutputs?: number
}

export interface ImageGenerateResult {
  url: string
}

export interface ImageProvider {
  id: string
  generate(options: ImageGenerateOptions): Promise<ImageGenerateResult | ImageGenerateResult[]>
}
```

### 4.4 视频生成

```typescript
export interface VideoGenerateOptions {
  imageUrl?: string
  prompt: string
  durationSeconds?: number
  aspectRatio?: string
}

export type VideoGenerateResult =
  | { url: string }
  | { taskId: string; statusUrl?: string }

export interface VideoProvider {
  id: string
  generate(options: VideoGenerateOptions): Promise<VideoGenerateResult>
}
```

## 5. Provider 实现清单（实际代码）

### 5.1 文本大模型

| 实现类 | `LLMProvider.id` | 依赖 | 适用 `provider` 配置 |
|--------|------------------|------|----------------------|
| `OpenAILLMProvider` | `"openai"` | `ai` 的 `generateText` + `@ai-sdk/openai` 的 `createOpenAI`（配置 `apiKey` / `baseURL`），经 `openai.chat(model)` 调用兼容 Chat Completions 形态的端点 | `openai`、`deepseek`、`anthropic`、`qwen`、`doubao`、`zhipu`、`custom-openai` |
| `GoogleLLMProvider` | `"google"` | `ai` + `@ai-sdk/google` 的 `createGoogleGenerativeAI` | `google` |

开发环境下 `OpenAILLMProvider` 会对请求/响应做 `console.log`，便于排查兼容接口问题。

### 5.2 图像 / 视频

| 实现类 | 说明 |
|--------|------|
| `PlaceholderImageProvider` | `id: "placeholder"`，使用 placehold.co 生成带文字的占位图，不调用外部 AI |
| `PlaceholderVideoProvider` | 占位实现，不调用外部 API |

DALL-E、ComfyUI、Runway 等在 `getImageProvider` / `getVideoProvider` 中留有注释占位，**未接线上逻辑**。

### 5.3 TTS

`lib/ai` 下 **无** `TTSProvider` 类型与实现；设置与测试逻辑在 API 层单独处理。

## 6. 配置读取与 Provider 获取

### 6.1 配置

> 路径：`lib/ai/config.ts`

- `getActiveConfig(type)`：读取 `Settings` 中 `activeLLMConfigId` 等；若无则取 `AIModelConfig` 中 `isDefault: true` 的同类型配置。
- `getLLMConfig` / `getImageConfig` / `getVideoConfig` / `getTTSConfig` 为兼容性别名，内部调用 `getActiveConfig`。

### 6.2 LLM：`lib/ai/llm/index.ts`

- `getLLMProvider()`：异步从数据库取当前 LLM 配置，并创建 Provider 实例；无配置时返回 `null`。
- `createLLMProviderFromConfig({ provider, apiKey, baseUrl, model })`：不读库，用于「测试连接」等场景。

路由逻辑概要：

```typescript
// 伪代码，与 lib/ai/llm/index.ts 一致
if (provider === "google") return new GoogleLLMProvider({ ... })
if ([openai, deepseek, anthropic, qwen, doubao, zhipu, custom-openai].includes(provider))
  return new OpenAILLMProvider({ apiKey, baseUrl, model })
return null
```

### 6.3 厂商元数据

> 路径：`lib/ai/providers.ts`

- `AI_PROVIDER_OPTIONS_BY_TYPE`：按 `llm` / `image` / `video` / `tts` 分组的厂商 label、模型下拉、`defaultBaseUrl`。
- `getProviderDefaultBaseUrl(provider)`：按厂商字符串解析默认 Base URL（测试连接时若用户未填则兜底）。

### 6.4 统一导出

> 路径：`lib/ai/index.ts`

对外导出类型、`getActiveConfig`、`getXxxConfig`、`getLLMProvider` / `getImageProvider` / `getVideoProvider` 以及统一调用入口 `callLLM` / `callImage`。

## 7. 各模块调用方式

### 7.1 统一层当前用途

- **设置页「测试连接」**：`POST /api/settings/ai-configs/test`  
  - LLM：调用 `createLLMProviderFromConfig` 后执行一次 `chat({ messages: [{ role: "user", content: "hi" }], maxTokens: 10 })`。  
  - 图像：ComfyUI 请求 `/system_stats`；其余尝试 `GET {baseUrl}/models` 校验 Key。  
  - TTS：`edge-tts` 直接成功；其余同 `models` 探测。

### 7.2 业务 API：环境变量 + `fetch`（与 `lib/ai` 并行存在）

以下路由在**未配置** `OPENAI_API_KEY` 时多使用本地占位生成；配置后通过 **`${OPENAI_BASE_URL}/chat/completions`** 调用（默认 `OPENAI_BASE_URL` 为 `https://api.openai.com/v1`，`OPENAI_MODEL` 默认 `gpt-4o-mini`）：

| 路由 | 说明 |
|------|------|
| `app/api/novels/[id]/analyze/route.ts` | 小说分析 |
| `app/api/episodes/generate-outlines/route.ts` | 分集大纲 |
| `app/api/scripts/generate/route.ts` | 剧本生成 |
| `app/api/script-shots/generate/route.ts` | 分镜生成 |
| `app/api/assets/generate/route.ts` | 资产提取等 |

上述路径 **尚未** 改为 `await getLLMProvider()` + `chat()`，因此用户在设置里切换 `AIModelConfig` **不会**自动作用于这些接口，除非后续改造为读取 `getActiveConfig("llm")` 或统一封装调用。

### 7.3 与文档旧版示例的差异

- 不存在 `src/lib/ai/factory.ts` 或 `AIProviderFactory.getLLM()`。
- 当前 `LLMProvider` **无** `generateTextStream`；流式剧本若需统一抽象，需在类型与实现中扩展。

## 8. 异步任务管理

设计文档早期版本中的 `Shot.imageTaskId`、`VideoClip` 等字段 **在当前 `prisma/schema.prisma` 中未实现**。图像/视频侧暂无真实异步任务与库表状态机；待接入真实厂商 API 时再补充 Schema 与轮询策略。

---

## 9. 错误处理与降级

代码中 **尚未** 实现统一的 `AIProviderError` 或 `withRetry` 工具。业务路由普遍在 `try/catch` 内打日志并 **回退到本地规则/占位内容**。后续可抽公共重试与错误分类，与 `lib/ai` 对齐。

---

## 10. 配置与多模型管理

### 10.1 数据库（Prisma）

`AIModelConfig` 与 `Settings`（含 `activeLLMConfigId` 等）与实现一致。`Settings` 中仍保留 **旧字段**（如 `textProvider`、`textModel`、`textApiKey`、`textBaseUrl` 等），与文档初稿一致，用于兼容或迁移；**新业务应以 `AIModelConfig` + active 指针为准**。

### 10.2 API

- `GET/POST /api/settings/ai-configs` — 列表与新建  
- `GET/PUT/DELETE /api/settings/ai-configs/[id]` — 单条读写删；默认配置会同步 `Settings` 中对应 `activeXxxConfigId`  

配置变更后无需手动清理缓存，`getLLMProvider`（及图像/视频同类函数）在每次调用时都会读取最新配置。
